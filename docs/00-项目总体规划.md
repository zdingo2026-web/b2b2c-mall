# B2B2C商城系统 - 项目总体规划

> 版本：v2.0 | 更新日期：2026-05-01

---

## 一、项目概述

### 1.1 项目定位

构建一套完整的B2B2C多商户商城系统，支持平台自营 + 多商户入驻模式，覆盖全终端用户触达。

### 1.2 四端架构

| 端 | 技术方案 | 目标用户 |
|---|---|---|
| 平台管理端 | SpringBoot + Vue3/Element Plus | 平台运营人员 |
| 商户管理端 | SpringBoot + Vue3/Element Plus | 入驻商户 |
| PC商城 | Vue3 + Nuxt3 (SSR) | PC端消费者 |
| 移动端商城 | UniApp (小程序/公众号/APP) | 移动端消费者 |

### 1.3 技术栈

| 层次 | 技术选型 |
|------|---------|
| 后端框架 | Java SpringBoot 2.7+ |
| 数据库 | MySQL 5.7 |
| 缓存 | Redis 6+ |
| 搜索引擎 | Elasticsearch 7.x |
| 消息队列 | RabbitMQ / RocketMQ |
| 对象存储 | 阿里云OSS / 华为云OBS |
| 前端PC | Vue3 + Nuxt3 + Pinia + TailwindCSS |
| 前端管理 | Vue3 + Element Plus + Vite |
| 移动端 | UniApp (Vue3) |
| 部署 | Docker + Nginx |

### 1.4 核心架构原则

1. **多租户隔离**：所有业务表携带 `tenant_id`（商户ID），平台级数据 tenant_id = 0
2. **统一API网关**：三端（平台/商户/C端）共用API层，通过认证体系区分身份
3. **订单引擎统一**：支持按商家自动拆单，自营与入驻商品共存
5. **支付网关抽象**：统一支付接口，支持多渠道扩展（微信/支付宝/银联/PayPal）
6. **营销引擎解耦**：营销活动与商品/订单解耦，通过策略模式扩展
7. **可视化装修引擎**：页面组件化，拖拽式配置，模板可导入导出

## 二、五期开发规划总览

| 期数 | 名称 | 周期 | 核心目标 |
|------|------|------|---------|
| 第一期 | MVP闭环 | 8-10周 | 四端最小可用版本，完成全链路闭环 |
| 第二期 | 通用功能 | 6-8周 | 营销活动、积分商城、店铺装修、三级分销 |
| 第三期 | 特色功能 | 5-6周 | 拍卖/夺宝/礼包/预售/社区团购/门店/白条 |
| 第四期 | 运营升级 | 4-5周 | 数据看板、智能推荐、搜索增强、IM客服 |
| 第五期 | 生态扩展 | 4-5周 | 开放平台、国际化、社交电商、SaaS化 |

## 三、Agent团队架构

### 3.1 团队总览

```
项目团队
├── 需求分析组
│   └── requirements-analyst    # 需求分析师 - PRD输出与评审
├── 架构设计组
│   ├── architect               # 架构师 - 系统架构设计
│   └── db-designer             # 数据库设计师 - 数据建模
├── 后端开发组
│   ├── backend-core            # 核心模块（用户/商品/订单）
│   ├── backend-trade           # 交易模块（支付/营销/分销）
│   └── backend-cms             # 内容模块（装修/搜索/通知）
├── 前端开发组
│   ├── frontend-pc             # PC商城网站
│   ├── frontend-mobile         # UniApp移动端
│   └── frontend-admin          # 管理后台前端
├── 质量保障组
│   ├── tdd-engineer            # TDD测试工程师
│   └── security-engineer       # 安全审查工程师
└── 运维部署组
    └── devops-engineer         # 运维部署工程师
```

### 3.2 各Agent职责

| Agent | 技能映射 | 核心职责 |
|-------|---------|---------|
| **requirements-analyst** | `planner`, `agent-harness-construction` | 每期需求分析、PRD编写与评审、功能优先级排序、需求变更管理、验收标准制定 |
| **architect** | `architect`, `planner` | 整体架构设计、技术选型、模块划分、API规范制定、多期架构演进规划 |
| **db-designer** | `architect`, `postgres-patterns` | 数据库表设计、索引优化、ER图输出、数据迁移方案 |
| **backend-core** | `springboot-patterns`, `springboot-security`, `java-coding-standards` | 用户体系、商品体系、订单体系、多租户框架 |
| **backend-trade** | `springboot-patterns`, `springboot-security` | 支付网关、营销引擎、分销佣金、积分系统 |
| **backend-cms** | `springboot-patterns`, `java-coding-standards` | 装修引擎、搜索服务、内容管理、消息通知 |
| **frontend-pc** | `frontend-patterns`, `frontend-design`, `web/coding-style` | PC商城全部页面、SSR、SEO、性能优化 |
| **frontend-mobile** | UniApp相关技能 | 小程序/公众号/APP页面开发、微信生态对接 |
| **frontend-admin** | `frontend-patterns` | 平台管理后台、商户管理后台、数据看板 |
| **tdd-engineer** | `tdd-workflow`, `springboot-tdd` | 单元测试/集成测试、覆盖率保障(80%+)、E2E测试 |
| **security-engineer** | `security-review`, `springboot-security` | 安全漏洞扫描、支付安全审计、权限体系审查 |
| **devops-engineer** | `docker-patterns`, `deployment-patterns` | CI/CD、容器化部署、数据库运维、监控告警 |

### 3.3 每期标准工作流

```
需求分析阶段（1-1.5周）
  requirements-analyst → 输出PRD → 评审通过
        ↓
架构设计阶段（0.5-1周）
  architect + db-designer → 架构方案 + ER图 → 评审通过
        ↓
并行开发阶段（4-6周）
  backend-core + backend-trade + backend-cms → 后端并行
  frontend-pc + frontend-mobile + frontend-admin → 前端并行
  tdd-engineer → 同步编写测试
        ↓
集成审查阶段（1-2周）
  security-engineer → 安全审查
  devops-engineer → 部署上线
  requirements-analyst → 验收确认
```

## 四、PRD输出与评审机制

### 4.1 PRD标准结构

每期PRD必须包含以下内容：

```markdown
# PRD - 第X期：XXX

## 1. 概述
- 本期目标
- 核心价值
- 依赖前置条件

## 2. 功能范围
- 功能清单（含优先级 P0/P1/P2）
- 各端功能矩阵
- 不包含的功能（明确边界）

## 3. 用户故事
- 角色 + 场景 + 期望结果

## 4. 业务流程
- 核心业务流程图
- 异常流程处理
- 状态流转图

## 5. 功能详细说明
- 每个功能点的详细描述
- 输入/输出/校验规则
- 边界条件和异常处理

## 6. 非功能需求
- 性能指标
- 安全要求
- 兼容性要求

## 7. 验收标准
- 每个功能的验收条件
- 测试场景覆盖

## 8. 里程碑
- 关键时间节点
- 交付物清单
```

### 4.2 评审流程

```
requirements-analyst 编写PRD
       ↓
architect 评审（技术可行性）
       ↓
后端/前端开发组评审（实现可行性）
       ↓
tdd-engineer 评审（可测试性）
       ↓
security-engineer 评审（安全性）
       ↓
评审通过 → 进入开发
评审不通过 → 修改后重新评审
```

### 4.3 PRD文件命名规范

## 六、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 多端并行开发接口不对齐 | 联调延期 | 第一周先定API契约，前端Mock开发 |
| 秒杀/抢购高并发 | 系统崩溃 | Redis预扣库存 + MQ异步下单 |
| 多租户数据泄露 | 安全事故 | SQL拦截器自动注入tenant_id + 单元测试覆盖 |
| 微信生态审核不通过 | 上线延期 | 提前了解审核规范，预留1周修改时间 |
| UniApp多端兼容性 | 开发效率 | 核心功能优先小程序，条件编译处理差异 |
