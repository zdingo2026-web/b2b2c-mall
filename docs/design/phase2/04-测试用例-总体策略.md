## 二、截图+AI分析测试模式

### 2.1 模式定义

传统测试依赖人工判断页面是否符合预期。本期采用「截图+AI分析」模式：

1. **执行操作** → 2. **截取屏幕截图** → 3. **AI分析截图** → 4. **判定通过/未通过**

AI分析截图时，依据每个测试用例中定义的「AI分析判断标准」进行判定。判断标准必须具体、可量化、可视觉验证。

### 2.2 截图要求

| 维度 | 要求 |
|------|------|
| 截图工具 | 管理端：浏览器DevTools截图或Playwright screenshot；移动端：设备截图/模拟器截图 |
| 截图范围 | 明确标注截取区域（整页/局部区域/弹窗/特定组件） |
| 截图时机 | 操作完成后立即截图，确保状态不被后续操作覆盖 |
| 截图命名 | `TC-{端}-{编号}-{步骤序号}.png`，如 `TC-P-COUPON-01-2.png` |
| 截图存储 | 统一存放至 `docs/design/phase2/screenshots/` 目录 |

### 2.3 AI分析判断标准

判断标准分为以下类型：

| 类型 | 示例 | 判定方式 |
|------|------|---------|
| 存在性 | "列表中存在>=5行数据" | 检测元素是否存在及数量 |
| 颜色值 | "剩余数量文字为红色(#E11148)" | 检测文字/背景颜色 |
| 文字内容 | "按钮显示'立即领取'" | OCR识别文字 |
| 布局结构 | "2列网格布局" | 检测元素排列方式 |
| 状态标识 | "Tag显示'进行中'且为绿色" | 综合文字和颜色 |
| 交互反馈 | "Toast提示'操作成功'" | 检测弹出层 |
| 空状态 | "列表显示'暂无数据'占位" | 检测空状态组件 |

### 2.4 通过/未通过判定规则

| 判定 | 条件 |
|------|------|
| **通过** | AI分析截图后，所有判断标准均满足 |
| **未通过** | 任一判断标准不满足（含列表页无数据） |
| **阻塞** | 前置条件不满足，无法执行测试步骤 |
| **跳过** | 当前环境不支持该测试（如小程序功能在H5环境） |

## 三、模拟数据策略

### 3.1 核心原则

> **列表页无数据 = 测试未通过**

所有涉及列表展示的页面，必须先通过API或数据库注入模拟数据，确保页面至少展示5条记录。截图验证时，AI需确认列表中存在数据行。

### 3.2 模拟数据分类

| 类别 | 说明 | 注入方式 |
|------|------|---------|
| 基础数据 | 会员、商家、商品、分类等一期已有数据 | 使用一期已有数据或API创建 |
| 二期新增数据 | 优惠券模板、秒杀活动、拼团活动、积分商品等 | 通过管理端API创建或直接插入数据库 |
| 关联数据 | 优惠券领取记录、分销订单、佣金记录等 | 通过C端操作产生或API模拟 |

### 3.3 模拟数据要求

1. **最少记录数**：每个列表页至少5条，关键列表至少10条
2. **数据多样性**：覆盖不同状态（如优惠券：启用/禁用/已结束）、不同类型（如满减/满折/满赠/无门槛）
3. **数据真实性**：名称、金额、日期使用合理值，不使用test/aaa等占位数据
4. **数据关联性**：关联ID必须真实存在，如优惠券的applySpuIds必须对应真实商品
5. **时间有效性**：进行中的活动时间范围需包含测试执行时间

### 3.4 模拟数据注入流程

```
1. 清理环境（可选）：DELETE FROM xxx WHERE tenant_id = 测试租户ID
2. 注入基础数据：调用一期API或使用已有数据
3. 注入二期数据：按各端测试用例文档中的模拟数据说明执行
4. 验证数据：调用查询API确认数据已正确入库
5. 执行测试：开始截图+AI分析测试流程
```

### 3.5 模拟数据示例格式

各端测试用例文档中，模拟数据以如下格式给出：

```json
{
  "接口": "POST /api/v1/platform/coupon/template",
  "数据": [
    {
      "couponName": "满100减20元优惠券",
      "couponType": 1,
      "couponValue": 20.00,
      "minAmount": 100.00,
      "totalCount": 1000,
      "limitPerUser": 1,
      "issuerType": 1,
      "applyScope": 0,
      "validType": 1,
      "validStartTime": "2026-05-01 00:00:00",
      "validEndTime": "2026-06-30 23:59:59"
    }
  ]
}
```

## 六、跨端测试协调

### 6.1 数据联动测试

以下场景需要跨端协调测试，确保数据一致：

| 场景 | 平台端操作 | 商户端验证 | 移动端验证 |
|------|-----------|-----------|-----------|
| 优惠券创建 | 创建平台券 | 可见该券 | 可领取该券 |
| 秒杀活动 | 创建活动+配置商品 | - | 可抢购 |
| 拼团活动 | - | 创建活动 | 可参团 |
| 积分商品 | 创建商品 | - | 可兑换 |
| 分销审核 | 审核分销商 | - | 审核结果更新 |
| 佣金提现 | 审核提现 | - | 到账通知 |
| 商家冻结 | 冻结商家 | 商家被冻结提示 | 店铺不可访问 |
| 红包发放 | 创建批次发放 | - | 收到红包 |

### 6.2 多租户隔离测试

| 测试点 | 方法 |
|--------|------|
| 优惠券隔离 | 商户A创建的券，商户B不可见 |
| 活动隔离 | 商户A的拼团活动，商户B不可见 |
| 分销数据隔离 | 商户A的分销订单，商户B不可见 |
| 积分商品非隔离 | 平台积分商品，全端可见 |
| 分销设置非隔离 | 平台统一设置，商户无法修改 |

### 6.3 一致性验证

以下数据项需在三端截图验证一致性：

| 数据项 | 验证方法 |
|--------|---------|
| 商品价格 | 同一商品在管理端和移动端截图对比 |
| 优惠券面值 | 管理端创建值与移动端展示值一致 |
| 积分余额 | 积分中心展示与后台查询一致 |
| 佣金金额 | 分销中心展示与后台查询一致 |
| 库存数量 | 秒杀库存/积分商品库存与后台一致 |
| 倒计时 | 基于服务端时间，三端同步 |

## 九、测试执行流程

```
┌─────────────────────────────────────────────────┐
│ 1. 环境准备                                      │
│    - 部署测试环境                                  │
│    - 准备测试账号                                  │
│    - 注入模拟数据（确保列表有数据）                   │
├─────────────────────────────────────────────────┤
│ 2. 执行测试（按端并行）                            │
│    - 平台管理端：按模块顺序执行                      │
│    - 商户管理端：按模块顺序执行                      │

│ 3. 截图采集                                       │

├─────────────────────────────────────────────────┤
│ 4. AI分析判定                                     │
│    - AI读取截图和判断标准                            │
│    - 输出通过/未通过/阻塞/跳过                       │
│    - 未通过用例记录原因和截图差异                     │
├─────────────────────────────────────────────────┤
│ 5. 跨端协调验证                                   │
│    - 数据联动测试                                   │
│    - 多租户隔离测试                                 │
│    - 一致性验证                                     │
├─────────────────────────────────────────────────┤
│ 6. 生成测试报告                                   │
│    - 用例执行统计                                   │
```
