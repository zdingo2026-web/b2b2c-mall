# 第二期：通用功能

> 版本：v1.0 | 更新日期：2026-04-30 | 预计周期：6-8周 | 前置依赖：第一期完成

## 一、本期目标

补齐主流B2B2C商城的通用功能，包括营销活动、积分商城、店铺可视化装修、三级分销体系，使商城具备完整的运营工具链。

### 1.1 核心原则

- **运营驱动**：本期功能以运营效率提升为核心目标
- **营销引擎化**：营销活动抽象为通用引擎，支持灵活组合
- **装修所见即所得**：可视化装修必须真实可用，非Demo级
- **分销合规**：三级分销严格控制在合法范围内

### 1.2 验收标准

- [ ] 运营人员可独立创建优惠券/秒杀/拼团/团购活动
- [ ] 消费者可参与营销活动并享受优惠
- [ ] 商户可自行装修店铺首页，预览后发布
- [ ] 分销商可申请成为分销员，生成海报推广
- [ ] 积分商城可正常兑换商品

## 二、需求分析师（requirements-analyst）工作内容

### 2.1 需求分析重点

1. 梳理各营销活动的业务规则和边界条件（如：优惠券叠加规则、秒杀库存扣减策略）
2. 定义分销佣金计算规则（三级分润比例、提现规则、结算周期）
3. 明确可视化装修的组件类型和属性配置
4. 定义积分获取与消费的完整规则体系
5. 分析营销活动与购物车/订单的交互逻辑

### 2.2 PRD输出要求

输出文件：`docs/prd/PRD-第2期-通用功能.md`

重点包含：
- 营销活动规则矩阵（各活动的触发条件/优惠力度/叠加规则/互斥规则）
- 分销佣金计算公式和结算流程
- 装修组件清单和属性配置表
- 积分获取/消费规则表
- 营销活动与购物车/结算页的交互流程图

### 2.3 评审要点

- 优惠券叠加规则是否清晰（多个优惠券能否同时使用）
- 秒杀库存扣减时的高并发处理方案
- 分销层级是否合法（不超过三级）
- 装修组件的数据结构是否可扩展

## 四、核心业务流程

### 4.1 营销活动与订单交互

```
用户选择商品 → 加入购物车
→ 购物车页面展示可用优惠券/凑单提示
→ 结算页自动计算最优优惠组合
→ 提交订单 → 优惠金额记录到订单
→ 退款时按比例退还优惠
```

### 4.2 优惠券叠加规则

```
平台券 + 商家券：可叠加
满减券 + 折扣券：同级别互斥
秒杀商品 + 优惠券：可配置是否允许
拼团 + 优惠券：不可叠加
```

### 4.3 秒杀库存扣减

```
活动开始前：预热库存到Redis
用户下单：Redis DECR 预扣库存
支付超时：Redis INCR 回补库存
支付成功：MySQL扣减实际库存
```

### 4.4 三级分销佣金计算

```
用户A → 通过分销员B的链接下单
→ B获得一级佣金（如10%）
→ B的上级C获得二级佣金（如5%）
→ C的上级D获得三级佣金（如2%）
→ 佣金在订单完成后进入可提现状态
→ 提现需平台审核
```

### 4.5 可视化装修流程

```
商户进入装修编辑器
→ 选择组件（轮播图/商品列表/广告位...）
→ 拖拽到画布、配置组件属性
→ 预览效果
→ 保存草稿 / 发布上线
→ 发布后C端首页即时更新
```

## 六、技术要点

### 6.1 营销引擎设计

```java
// 策略模式 - 统一营销活动接口
public interface PromotionStrategy {
    PromotionResult calculate(OrderContext context);
    boolean isAvailable(OrderContext context);
}

// 实现
CouponPromotion   → 优惠券策略
SeckillPromotion  → 秒杀策略
GroupBuyPromotion → 拼团策略
GrouponPromotion  → 团购策略
DiscountPromotion → 折扣策略
```

### 6.2 装修引擎设计

```
页面 = 模板 + 组件列表
组件 = {
  type: "banner" | "goods_list" | "ad_slot" | "rich_text" | ...
  props: { ... }    // 组件属性
  sort: 1           // 排序
  visible: true     // 是否显示
}
→ JSON存储，C端渲染时按sort顺序加载组件
```

### 6.3 秒杀高并发方案

- Redis预热库存 + Lua脚本原子扣减
- MQ异步下单，前端轮询结果
- 接口限流（令牌桶）
- 动态URL防止提前抢购
