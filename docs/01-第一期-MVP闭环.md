# 第一期：MVP闭环

> 版本：v1.0 | 更新日期：2026-04-30 | 预计周期：8-10周

---

## 一、本期目标

完成B2B2C商城全链路最小可用版本，四端均能独立运行，实现 **"浏览→加购→下单→支付→发货→收货→评价"** 完整闭环。

### 1.1 核心原则

- **先通后优**：优先保证链路通畅，非核心体验可迭代优化
- **架构先行**：多租户、统一支付、统一订单等底层架构必须到位，为后续功能留好扩展点
- **API契约驱动**：前后端并行开发，先定接口再实现

### 1.2 验收标准

- [ ] 消费者可在PC/移动端完成：注册→登录→浏览商品→加购物车→下单→支付→查看订单→确认收货→评价
- [ ] 商户可在商户端完成：登录→发布商品→管理订单→发货→查看收入
- [ ] 平台可在管理端完成：商家入驻审核→商品审核→订单管理→会员管理→基础数据查看

## 二、需求分析师（requirements-analyst）工作内容

### 2.1 需求分析重点

1. 梳理MVP最小功能集，明确"必须有"与"可以没有"的边界
2. 编写用户故事，覆盖三种角色（消费者/商户/平台管理员）的核心场景
3. 定义业务流程：注册登录流程、下单支付流程、商家入驻流程、订单状态流转
4. 明确验收标准：每个功能的可测试验收条件
5. 识别跨端一致性需求：PC端与移动端功能对齐策略

### 2.2 PRD输出要求

输出文件：`docs/prd/PRD-第1期-MVP闭环.md`

必须包含：
- 功能清单及优先级（P0必须/P1重要/P2可选）
- 四端功能矩阵（哪些功能在哪个端实现）
- 核心业务流程图（至少覆盖：下单流程、支付流程、商家入驻流程、订单状态流转）
- 各功能详细说明（输入/输出/校验规则/异常处理）
- 验收标准（每个功能的可测试条件）

### 2.3 评审要点

- 功能边界是否清晰（不做哪些功能）
- 用户故事是否覆盖核心路径
- 业务流程是否有遗漏的异常分支
- 验收标准是否可执行可验证

## 四、核心业务流程

### 4.1 订单状态流转

```
待付款 → 待发货 → 待收货 → 已完成
  ↓         ↓        ↓
已取消   退款中    退款中
                    ↓
                 已退款
```

### 4.2 多商家拆单流程

```
购物车（含A店+B店商品）→ 结算页 → 按商家拆分为子订单
→ 子订单A（A店商品）+ 子订单B（B店商品）
→ 各自独立发货、独立物流
```

### 4.3 商家入驻流程

```
商家提交入驻申请（企业信息/营业执照/联系人）
→ 平台审核（通过/拒绝）
→ 审核通过：创建商户账号，发送通知
→ 商户登录后台，完善店铺信息
→ 开始发布商品
```

## 五、架构设计重点（architect工作内容）

### 5.1 多租户架构

- 所有业务表增加 `tenant_id` 字段
- 平台级数据 `tenant_id = 0`
- MyBatis-Plus 拦截器自动注入租户条件
- 超级管理员可跨租户查询

### 5.2 统一认证体系

```
平台管理员 → JWT Token (role=platform)
商户管理员 → JWT Token (role=merchant, tenantId=xxx)
消费者     → JWT Token (role=member)
```

### 5.3 统一订单引擎

- 主订单 + 子订单结构（支持多商家拆单）
- 订单状态机（状态流转引擎，可配置化）
- 订单快照机制（商品价格/信息快照，防止后续变更影响历史订单）

### 5.4 统一支付网关

```
PayService（抽象接口）
├── WechatPayService
├── AliPayService
└── BalancePayService
```
- 统一支付/退款接口
- 支付回调统一处理
- 支付流水记录

### 5.5 商品体系

```
SPU（标准产品单元）
├── 商品名称、描述、分类、品牌
└── SKU（库存量单元）
    ├── 属性组合（颜色+尺寸）
    ├── 价格、库存、图片
    └── 编码
```

## 六、数据库设计重点（db-designer工作内容）

### 6.1 核心表清单

| 领域 | 核心表 |
|------|--------|
| 用户 | member, member_address, member_auth（第三方绑定） |
| 商户 | tenant, tenant_category, tenant_settle, tenant_admin |
| 商品 | product_spu, product_sku, product_category, product_brand, product_image |
| 订单 | order_main, order_item, order_address, order_log, order_snapshot |
| 支付 | payment_record, payment_refund |
| 营销 | coupon, coupon_record（预留表，第二期启用） |
| 系统 | sys_admin, sys_role, sys_permission, sys_config, region |

### 6.2 设计原则

- 所有表包含 `tenant_id`（系统表除外）
- 所有表包含 `create_time`, `update_time`, `deleted`（逻辑删除）
- 金额字段统一使用 `DECIMAL(10,2)`
- 订单号使用雪花算法生成
- 外键约束在应用层实现，不使用数据库外键

### 7.2 后端并行分工

```
backend-core:
  - 用户体系（注册/登录/OAuth/JWT）
  - 商品体系（SPU/SKU/分类/品牌）
  - 订单体系（下单/拆单/状态机/快照）
  - 多租户框架（拦截器/数据隔离）

backend-trade:
  - 支付网关（微信/支付宝/余额）
  - 支付回调处理
  - 退款流程
  - 购物车服务

backend-cms:
  - 文件上传服务（OSS）
  - 短信服务
  - 搜索服务（MySQL LIKE，预留ES扩展）
  - 基础配置服务
```

### 7.3 前端并行分工

```
frontend-pc:
  - 首页、分类页、列表页、详情页
  - 购物车、结算、支付
  - 会员中心

frontend-mobile:
  - 小程序首页、分类、搜索
  - 商品详情、购物车、下单支付
  - 会员中心

frontend-admin:
  - 平台管理端全部页面
  - 商户管理端全部页面
```
