# 第四期：运营效率与体验升级

> 版本：v1.0 | 更新日期：2026-04-30 | 预计周期：4-5周 | 前置依赖：第三期完成

---

## 一、本期目标

提升运营效率和用户体验，补齐管理工具链和智能化能力，包括数据看板、智能推荐、搜索增强、IM客服、内容运营、消息通知、后台提效等功能。

### 1.1 核心原则

- **数据驱动**：以数据看板支撑运营决策
- **智能推荐**：基于用户行为的个性化推荐
- **效率至上**：后台批量操作减少人工成本
- **即时沟通**：IM客服缩短用户决策周期

### 1.2 验收标准

- [ ] 平台/商户可查看实时运营数据看板
- [ ] "猜你喜欢"推荐结果与用户行为相关
- [ ] 搜索支持拼音/智能分词/品牌价格区间筛选
- [ ] IM客服可实时沟通，支持图文消息
- [ ] 后台支持批量导入导出、一键发货等提效操作
- [ ] 短信/邮件/站内信三通道消息通知正常

## 二、需求分析师（requirements-analyst）工作内容

### 2.1 需求分析重点

1. 数据看板的指标体系设计（GMV/UV/转化率/客单价/复购率等）
2. 智能推荐算法的输入特征和推荐策略
3. 搜索增强的功能边界（分词/拼音/同义词/纠错）
4. IM客服的消息类型和会话管理规则
5. 消息通知的触发场景和模板设计
6. 后台批量操作的边界条件（导入格式/数量限制/失败处理）

### 2.2 PRD输出要求

输出文件：`docs/prd/PRD-第4期-运营升级.md`

重点包含：
- 数据看板指标清单和计算公式
- 推荐算法策略说明（协同过滤/热门/关联推荐）
- 搜索增强功能矩阵
- IM客服功能规格（消息类型/会话管理/客服分配）
- 消息通知场景-模板-通道映射表
- 批量操作规格（导入格式/校验规则/错误处理）

### 2.3 评审要点

- 数据看板的实时性要求（秒级/分钟级/小时级）
- 推荐算法的冷启动问题
- IM消息的存储和合规（聊天记录保存）
- 消息通知的频率控制（防骚扰）
- 批量导入的事务性和幂等性

## 四、核心业务流程

### 4.1 智能推荐流程

```
用户浏览/购买 → 行为日志采集
→ 离线计算用户画像标签
→ 实时请求推荐接口
→ 返回个性化商品列表
→ 冷启动用户：热门/新品/同分类推荐
```

### 4.2 搜索增强流程

```
用户输入关键词
→ 拼音转换 + 分词 + 同义词扩展 + 纠错
→ ES多字段检索（标题/描述/分类/品牌）
→ 聚合品牌/价格区间
→ 返回结果 + 筛选面板
```

### 4.3 IM客服流程

```
消费者点击客服 → 分配在线客服
→ WebSocket实时通信
→ 支持文字/图片/商品卡片/订单卡片
→ 客服离线 → 留言模式
→ 历史消息可查
```

### 4.4 消息通知流程

```
业务事件触发（如：商家审核通过）
→ 查询通知配置（该场景需发哪些通道）
→ 渲染消息模板
→ 发送短信/邮件/站内信
→ 记录发送日志
```

## 六、技术要点

### 6.1 Elasticsearch集成

- MySQL Binlog → Canal → ES 数据同步
- 索引设计：商品索引（标题/描述/分类/品牌/价格/销量）
- 拼音分词器：pinyin analyzer
- 搜索建议：completion suggester

### 6.2 推荐引擎

- 协同过滤：基于用户行为的商品相似度计算
- 实时推荐：Redis缓存用户画像标签
- 离线计算：定时任务更新推荐结果
- 冷启动：热门/新品/分类兜底策略

### 6.3 IM客服

- WebSocket长连接
- 消息存储：MongoDB（聊天记录）
- 消息推送：离线消息通过MQ推送
- 文件上传：OSS存储图片

### 6.4 数据看板

- 实时数据：Redis计数器（今日UV/订单/支付）
- 历史数据：定时聚合到统计表
- 大屏展示：WebSocket推送实时更新
